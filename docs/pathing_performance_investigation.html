What production A* actually does

Endpoints move from top-left to centers. Elevation is explicit or inferred; the higher endpoint elevation is chosen, then the nearest layer. After a direct-line fast path, the implicit lattice uses clearance-aware point/edge tests against indexed wall segments. A greedy pass keeps the farthest visible waypoint.

Fail-open semantics

Every planning failure returns straight [start, end]—including a cap hit, missing connection, or blocked search. That preserves motion but can visibly cross walls. There is no agent-agent avoidance or dynamic-obstacle routing, and cost grows roughly linearly with moved-agent count.

04 · Correcting the animation assumption

Raycasting can repeat while the overlay moves

A* is absent during animation. View-cone work is not: preview cones can rebuild and synchronously raycast every animation tick.

Target-shaped cone versus moving-origin cone

Scrub the transition. Walls make the correctness difference visible.

Ray origin mode
35%
stored target origin displayed/computed origin walls visibility polygon

Performance implication

VisionPolygon.compute runs synchronously in ViewConeWidget.build when origin and geometry resolve. It queries nearby indexed segments, emits regular rays at ≤2° plus endpoint/range-circle events and ±epsilon corner rays, intersects candidates, and sorts the result. Approximate complexity: O(A log A + A·S). There is no visibility-polygon memoization.

Correctness implication

Today a moving preview raycasts from the stored target world origin. The resulting destination-shaped polygon is translated along the overlay path, so its occlusion shape does not correspond to intermediate world positions. Recomputing at the interpolated origin would be geometrically correct—but may increase already-repeating work unless inputs are memoized or behavior is deliberately frozen.

05 · Coordinate-space stack

There is no separate “mount-boundary” subsystem

Four layers explain the layout and search space. Resize and zoom change presentation; normalized geometry and path assets stay cached.

What changes on resize

LayoutBuilder updates CoordinateSystem and recenters presentation. There is no DOM getBoundingClientRect equivalent and no measurement of rendered map obstacles.

What does not change

Normalized SVG/path geometry remains reusable. A* searches within the outer SVG contour axis-aligned bounding box—not the viewport, the centered map rectangle, or a separately measured mount boundary.

Practical reading: responsive layout work can affect rendering cost, but viewport size is not rebuilding the normalized navigation geometry or redefining its search bounds.

08 · Ranked plan

Measure first; optimize stable geometry second

Ordering matters: each structural change should answer a trace-backed question.

01

Instrument profile/release traces

Bracket asset load/decode, UI-isolate geometry phases, per-agent planning (expansions and containment calls), overlay build/sort/publication, and cone computation. Compare cold/warm, direct/difficult routes, and 1 / several / many moved agents. Record ray count, candidate segment count, cone rebuild count, fallback reason, and frame build/raster health.

Cache floor containment or precompute a walkable lattice

Key immutable data by map / side / elevation layer / clearance bucket. Start with containment acceleration or lattice metadata; validate memory, cold-build movement, and route quality against trace baselines.

Define transition-cone semantics, then memoize

Either memoize polygons on complete geometry inputs or intentionally freeze them during motion. If correctness requires moving-origin raycasts, include interpolated origin in the key and measure frame cost. Add transition-ray-origin tests before changing behavior.

Add request cancellation by generation token

Tag page requests and ignore stale hydration, snapshots, planning, and animation starts when a newer generation wins. This is primarily correctness protection with wasted-work benefits.

Evaluate first-use SVG parsing/classification or route batches as self-contained payloads. Measure serialization and transfer overhead; shared object graphs do not cross isolates for free.

Static-grid tradeoff

A precomputed grid can make warm planning faster and containment simpler. It also adds asset size and versioning, clearance/elevation/side variants, and route coarseness. Prefer a hybrid: keep exact authoritative contours and precompute immutable geometry/lattice metadata. Raw route caching is a weaker first direction because endpoints, clearances, sides, elevations, and asset versions create a large correctness-sensitive key space.
